Escrow contract

Flow order — setup then per-verification

①Group owner DMs Canvas AI → shares group link + payout wallet → written to Postgres group registry → bot added as admin to Telegram group
②Advertiser DMs Canvas AI → selects group by topic → Canvas AI shows current top bid for that group → advertiser sets bid per verification above the current top → Canvas AI calls Bankr API → USDC locked at bid price × quantity in escrow on Base · if a new advertiser outbids them later Canvas AI notifies the displaced leader via Telegram and prompts them to rebid
③Human attempts to join group → Telegram Bot API fires webhook to Canvas AI → bot queries Postgres for highest active bid on that group → selects that advertiser's task
④Bot mutes user in group → sends captcha task in private DM · task is matched to group topic and advertiser category
⑤Human completes task and replies via DM → bot receives response through Telegram webhook → response stored in Postgres verification queue
⑥Canvas AI calls Base directly (RPC) → step ① contract call logs completion attempt onchain · advertiser can see attempt count at any time · budget not yet released · no Bankr involvement
⑦Bot sends response to Kimi inference → Kimi API scores it as genuine or random · if score passes threshold step 2 fires · if not human is re-prompted or denied
⑧Canvas AI calls Base directly (RPC) → step ② contract call releases USDC from escrow to group owner wallet at the winning bid price · budget draws down · payout is immediate and non-custodial · no Bankr involvement
⑨Bot unmutes human and sends group invite link via DM · human is now a member · Telegram API handles the group admission
⑩Advertiser notified via Canvas AI · completion data delivered · response included · 99 verifications remaining in budget
