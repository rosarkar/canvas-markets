## Problem Statement

Telegram group owners do the work of gating their communities — using bots like Rose to verify new members — and earn nothing from it. KOLs running 1,000–10,000 member DeFi groups on Base earn real money from Twitter (sponsored posts, shilling deals) but have zero monetisation mechanism for Telegram, where they actually spend most of their time.

On the other side, DeFi protocols (Moonwell, Avantis) need to reach verified crypto-native users for acquisition and intent data. The current method is organic relationships and paid KOL shilling — expensive, hard to measure, impossible to scale beyond your personal network.

Canvas AI is the monetisation layer that connects both sides: group owners earn per verified join; advertisers pay for verified, intent-signalled users entering relevant communities. The join-verification moment — already happening via Rose — becomes a two-sided marketplace.

Telegram groups — two-way communities — have nothing. This is not a gap in a crowded market. It is a gap nobody has named.

## Constraints

- Pre-build. Core architecture is v0.2. Open questions in this doc must be resolved before those specific components are built — they do not block other components starting.
- Tech stack: Telegram Bot API (webhook, bot as group admin), Railway (always-on agent server), Postgres (state), Bankr API (advertiser payment UX + USDC deposit), Base direct RPC (step ① log, step ② payout), Kimi API (response scoring).
- **Step ① (log):** fires when a human submits a captcha response. Canvas AI calls the escrow contract to record a completion attempt onchain. Budget is NOT released. Advertiser can read attempt counts on Base at any time. Step ① is an audit log — it records every attempt regardless of Kimi score. Its purpose is advertiser transparency (attempt volume visible onchain without querying Canvas AI) and payout dispute resolution. It does not gate admission and is not used for deduplication (cooldown is enforced in Postgres, not onchain).
- **Step ② (payout):** fires only when Kimi scores the response above the pass threshold. Canvas AI calls the escrow contract to release the winning bid price in USDC to the group owner's registered wallet. Budget draws down by one verification.
- **Payout calculation:** group owner receives 100% of the winning bid price for v1. Platform fee is $0 at launch to optimise for supply-side adoption. Revisit at 500 verifications.
- **Bid auction mechanics (v1):** first-price auction — highest active bid wins each join. No minimum bid increment. Concurrent bids: last-write-wins on Postgres `bid_per_verification` field (acceptable race at POC scale). In a disputed payout, `bid_log` is the source of truth — `advertiser_budgets.bid_per_verification` is a cached convenience field; `bid_log` is append-only and immutable. Minimum bid: $0.10 (floor).
- Smart contract on Base: non-custodial USDC escrow, per-verification payout at winning bid price. **Pre-launch requirement: lightweight contract audit before any real USDC is deposited. No mainnet launch without this.**
  - **Escrow deposit model (v1):** single shared contract. All advertisers deposit into the same contract. Each advertiser's balance is tracked by `(advertiser_id, group_id)` key inside the contract mapping. This is simpler to deploy and audit than one contract per campaign. Refunds return to the depositing address stored per advertiser entry. The contract is not upgradeable for v1 — simpler to audit.
  - **USDC decimal representation:** USDC on Base uses 6 decimal places. All bid amounts are stored as integer USDC microunits throughout — in Postgres, in the contract, and in all RPC calls. 1 USDC = 1,000,000 units. $0.10 = 100,000. $0.35 = 350,000. No float arithmetic anywhere in the payout path. Postgres column type: `BIGINT`. Contract parameter type: `uint256`. Display layer (Canvas AI DM messages) converts to dollars for readability.
  - **Minimum contract function signatures (to be fully specified during contract design):**
    - `depositBudget(advertiserId, groupId, bidPerVerification, quantity)` — locks `bidPerVerification × quantity` USDC. Emits `BudgetDeposited(advertiserId, groupId, amount)`.
    - `logAttempt(sessionId, groupId, advertiserId)` — records completion attempt. Emits `AttemptLogged(sessionId)`.
    - `releasePayout(sessionId, ownerWallet, amount)` — releases `amount` USDC to `ownerWallet`. Emits `PayoutReleased(sessionId, ownerWallet, amount)`. Callable only by the relayer key.
    - `refund(advertiserId, groupId)` — returns unused balance to depositing address. Callable by advertiser or relayer.
  - **Audit scope:** single experienced reviewer (not a Big 4 firm). Checklist: (a) USDC can only flow to wallets in the `groups` table — no arbitrary destination, (b) daily spend cap enforced onchain, (c) refund returns to depositing address per advertiser entry, (d) no reentrancy on payout, (e) no owner key can drain the contract unilaterally. Estimated: 1–2 days, $500–$2,000. Options: Sherlock crowdsourced, Code4rena, or a trusted individual auditor from the Base ecosystem. Founder to identify reviewer before smart contract is written.
- **Signing key risk:** agent server holds the Base relayer key. Mitigations before launch: (a) daily spend cap enforced onchain (see contract audit checklist), (b) relayer wallet is a separate key with no other permissions and holds only enough ETH for ~500 transactions at a time, (c) rate-limit: maximum 60 payouts per hour per group (step ② calls), (d) step ① calls are not individually rate-limited but the 5-minute TTL per verification and 24-hour cooldown per user create a natural bound. Relayer ETH balance: funded manually to ~0.01 ETH; Canvas AI logs a warning to the founder's DM when balance drops below 0.002 ETH. This is acknowledged as the primary security surface.
- **Bankr deposit TTL:** when an advertiser confirms they are depositing (direct Base fallback or Bankr flow), Canvas AI begins polling. If no deposit is confirmed within 2 hours, the pending campaign record is marked `campaign_status = 'expired'` and the advertiser is notified via DM. The Postgres record is not deleted — it serves as an audit trail. The advertiser can reinitiate.
- Bankr integration: Canvas AI uses Bankr's skills for the wallet/payment UX layer. **Pre-build spike required:** confirm Bankr's API supports (a) USDC deposits to a specified contract address, (b) advertiser wallet creation via agent conversation, (c) webhook callbacks on payment confirmation. If Bankr does not support this flow, the payment architecture defaults to a direct Base wallet flow with no Bankr dependency. **Direct Base fallback flow:** Canvas AI presents the advertiser with the escrow contract address and the exact USDC amount to send. Canvas AI monitors the contract for deposit events via RPC polling (every 30 seconds). On deposit confirmed: updates `remaining_budget` in Postgres, activates the bid. No conversational payment UX. Advertiser must hold USDC in a self-custody wallet. This is acceptable for v1 — Moonwell and Avantis have treasury teams that can initiate manual transfers.
- No existing codebase. Greenfield.

## Recommended Approach: Full Canvas Build (v0.2)

Build the complete system as designed, with one non-negotiable constraint: **Moonwell and Lennox Cartel conversations happen this week, in parallel — not after launch.**

The founder chose the full build over the concierge approach because of their direct relationships on both sides. That's the right call given the context. But the build does not replace the conversations. Those conversations validate pricing, pitch, and the advertiser budget assumption before the escrow layer is live.

**Build sequence:**

1. **Bankr API spike + Kimi API spike** — both before any other build. Run in parallel.
   - **Bankr spike:** confirm (a) USDC deposits to a specified contract address, (b) advertiser wallet creation via agent conversation, (c) webhook callbacks on payment confirmation. This is the highest-risk dependency. Result: either proceed with Bankr or default to direct Base wallet (see fallback flow above).
   - **Kimi spike:** confirm (a) API accepts a text response and returns a quality score, (b) latency is under 3 seconds for a short paragraph, (c) cost per call is under $0.01. If Kimi fails: fallback is a synchronous keyword presence check — does the response contain ≥2 domain-specific terms from a curated word list? Lower quality signal but no external dependency and zero latency. Implement keyword fallback regardless; Kimi is the default but the fallback must be live before launch. **Keyword list ownership:** founder writes the initial DeFi keyword list before first campaign launch (example terms: defi, liquidity, yield, pool, wallet, staking, lend, borrow, bridge, swap, base, usdc, eth — ~20–30 terms). Stored as a config file in the repo, not hardcoded. Expandable per group topic.

2. **Telegram bot** — join intercept, DM deep-link (required: user must tap a link to start bot conversation before the bot can DM them — send a button in the group welcome message: `t.me/canvasbot?start=verify_<session_token>`), captcha delivery, response handling, admit/kick.
   - **Muting note:** Telegram's restriction API applies only after the user has sent at least one message in the group. For new joins who say nothing, the flow is: join → immediately send a welcome message with the deep-link button → if no DM response within 5 minutes → kick (not mute). Muting applies only for users who engage in the group before completing verification.
   - **Welcome message trigger:** on `chat_member` update with `new_chat_member.status = 'member'`. The bot sends a `sendMessage` to the group with an inline keyboard button containing the deep-link. If `sendMessage` fails (bot not admin, rate limited, etc.): log error, do not attempt to kick or mute, notify founder's DM: "Bot failed to post welcome message in [group name] — check admin status."
   - **Bot admin loss detection:** on any `sendMessage`, `banChatMember`, or `restrictChatMember` call, if Telegram returns error 400 "not enough rights": pause the campaign for that group, mark `is_active = false` in `groups`, notify the group owner via Canvas AI DM: "Canvas AI lost admin access to [group]. Please re-add the bot as admin and reply 'resume' to restart."
   - **Session token:** deep-link token is a 128-bit random UUID generated per join event, stored in `verifications.verification_id`. Single-use: once a DM is received with a given token, it is marked consumed. Expiry matches the verification TTL (5 minutes).

3. **Postgres schema** — with explicit verification state machine:
   - States: `PENDING → DEEP_LINK_SENT → TASK_SENT → RESPONSE_RECEIVED → STEP1_FIRED → STEP1_FAILED → SCORING → KIMI_FAILED → PASSED → FAILED → TIMED_OUT`
   - `STEP1_FAILED`: Base RPC call failed or contract reverted. Bot sends user a DM: "Something went wrong on our end. Please try again in a few minutes." User is kicked from the group. The 24-hour cooldown does NOT apply to STEP1_FAILED — it is a system error, not a failed attempt. User can rejoin and start again immediately.
   - `KIMI_FAILED`: Kimi API returned error or timed out. Fallback: human reviewer queue (founder reviews manually for first 50 responses). If not reviewed within 10 minutes: admit user, mark PASSED (optimistic fallback for launch period).
   - `MUTED`: user has sent a message in the group before completing DM verification (Telegram restriction API is applicable). Transition: `PENDING → MUTED → DEEP_LINK_SENT`.
   - **Bid price lock-in:** locks at the moment `TASK_SENT` is written. If a higher bid is placed while a verification is in-flight, the in-flight verification completes at the locked price — the displaced advertiser's task is served and they pay out if Kimi passes. The displacement only affects the next join. The outbid notification is sent to the displaced advertiser at the moment the new bid lands, regardless of any in-flight verifications.
   - TTL: 5-minute expiry on DEEP_LINK_SENT and TASK_SENT states. On expiry: kick user (or unmute if MUTED), mark TIMED_OUT. TIMED_OUT triggers the 24-hour cooldown — it counts as an attempt.
   - **24-hour cooldown:** starts from the `TIMED_OUT`, `FAILED`, or `PASSED` state transition (whichever applies). A user who re-joins the group while in cooldown is immediately kicked with bot DM: "You already attempted verification for this group recently. Try again in [remaining hours]."
   - Tables:
     - `groups` (group_id, tg_group_id, owner_wallet, owner_tg_id, registered_at, verification_task_text [the fallback generic task for this group — set by founder at registration, editable via Canvas AI], is_active BOOLEAN)
     - `advertiser_budgets` (advertiser_id, group_id, bid_per_verification BIGINT [microunits], remaining_budget BIGINT [microunits], advertiser_tg_id, campaign_status ENUM['active','paused','exhausted'], outbid_notified BOOLEAN)
     - `verifications` (verification_id UUID PRIMARY KEY, tg_user_id BIGINT, group_id, advertiser_id, state, locked_bid_price BIGINT [microunits], kimi_score INTEGER, response_text TEXT, created_at TIMESTAMPTZ, updated_at TIMESTAMPTZ, expires_at TIMESTAMPTZ, attempt_count INTEGER DEFAULT 1)
     - `bid_log` (log_id SERIAL PRIMARY KEY, advertiser_id, group_id, bid_amount BIGINT [microunits], placed_at TIMESTAMPTZ — append-only, never updated)

   - **Telegram bot admin permissions required:** Ban Users (for kick), Restrict Members (for mute), Delete Messages (to clean up deep-link messages), Invite Users via Link. Configure via @BotFather when adding bot as admin.

4. **Canvas AI agent (Telegram DM bot)** — buy flow (Bankr API for USDC), register flow, live bid display ("Current top bid: $0.28 — DeFi AI Lab"), outbid notifications.
   - **Group owner registration conversation (minimum viable flow):**
     1. "What's your group's Telegram link?" → validate it's a real group (bot must already be admin); if not: "Add me as admin first, then send the link."
     2. "What's your Base wallet address for payouts?" → validate EVM address format (starts with 0x, 42 chars); if invalid: "That doesn't look like a Base address — try again."
     3. "What's the verification question your community will see when no advertiser is running? (e.g. 'In one sentence: what do you use DeFi for?')" → store as `verification_task_text`.
     4. Confirm summary → write to `groups` table.
   - **Advertiser registration + buy conversation (minimum viable flow):**
     1. "Which groups do you want to target? I can show you DeFi, trading, or yield groups." → display matching groups with current top bids.
     2. "How many verifications do you want to buy?" → parse integer; minimum 10.
     3. "What's your bid per verification? Current top bid on [group] is $X.XX. Bid above that to win each join." → parse dollar amount; validate ≥ $0.10 and > current top bid.
     4. Confirm: "You're locking $[total] USDC for [quantity] verifications at $[bid] each. Confirm?" → on confirm: initiate Bankr deposit or show escrow address.
     5. On deposit confirmed: "You're live. I'll message you daily with results and alert you if you're outbid."
     - **Input parsing for bid amounts:** accept "$0.35", "0.35", ".35", "35 cents". Reject anything that doesn't parse to a valid dollar amount. If exactly equal to current top bid: "You need to bid above $X.XX — try $[current + $0.01]?"

5. **Base smart contract** — USDC escrow, step ① completion log, step ② payout. Audit before mainnet.

6. **Bankr API integration** — advertiser wallet UX, USDC deposit. (Depends on spike in step 1.)

7. **Bidding marketplace** — live top bid in buy flow, outbid notification queue, bid ladder in Postgres. **Conscious scope bet:** this is pre-second-advertiser complexity. Included because founder chose the full build. If Moonwell is the only advertiser at launch, this layer has zero effect on day-1 behaviour.
   - **Outbid notification delivery:** when a new bid is placed and it beats the current top bid, Canvas AI immediately reads `advertiser_tg_id` for the displaced leader from `advertiser_budgets` and sends a Telegram DM via the bot. Message: "You've been outbid on [group name]. New top bid: $X.XX. Your bid was $Y.YY. Your task is no longer being served. Reply with a number above $X.XX to reclaim the top slot." If `advertiser_tg_id` is null (advertiser registered before this field existed), skip notification and log a warning. Delivery is best-effort: if the DM fails (user blocked the bot), mark `outbid_notified = false` in `advertiser_budgets` and do not retry — the advertiser will see their status next time they DM Canvas AI.

8. **Kimi API integration** — response scoring. **Conscious scope bet:** no observed evidence of fake response rates yet. For the first 5 groups, manual review is a viable fallback if Kimi integration is slow. Pass threshold: calibrate from first 50 real responses before hardcoding.
   - **Kimi input format:** send both the task question and the user's response. System prompt (v1 draft): "You are scoring a human verification response. The user was asked a question to prove they are a genuine member of a DeFi community, not a bot or spam account. Score their response from 0 to 100: 100 = clearly genuine and thoughtful, 0 = empty, random characters, or obviously evasive. Return only a JSON object: {\"score\": <integer 0-100>}." User message: "Question: [task_text]\n\nResponse: [response_text]". The system prompt and task text are refined during calibration.
   - **Calibration process:** founder manually reviews first 50 real responses in a shared spreadsheet (columns: response text, admit/deny decision, reasoning). Kimi threshold is set as the lowest Kimi score that correctly classified ≥90% of founder-reviewed passes with ≤5% false negatives. Threshold written as a named constant in the codebase (`KIMI_PASS_THRESHOLD`) before second campaign launches. Do not hardcode before observing real data.

**Observability without a web UI:** Advertiser can see step ① logs on Base block explorer (attempt counts are onchain). For a non-technical growth team at Moonwell, Canvas AI bot delivers a daily completion report via Telegram DM at 9am UTC. No web dashboard for v1.
- **Daily report delivery:** Railway cron job at 09:00 UTC, queries `advertiser_budgets` for all `campaign_status = 'active'` records, sends one DM per advertiser per active campaign via the bot using `advertiser_tg_id`. If `advertiser_tg_id` is null, skip silently and log warning. If an advertiser has no completions that day, still send report ("0 verifications today — campaign is live and waiting").
- **Daily report content:** (1) verifications completed today / total, (2) USDC paid out today / total, (3) USDC remaining in budget with a "running low" alert at ≤20% remaining, (4) current top bid status (are you still the top bidder?), (5) up to 3 anonymised response summaries from that day (intent signal: what users said they'd do). Format: plain-text Telegram message, no links, no attachments — readable on mobile without tapping through.
- **Security — webhook validation:** the Telegram webhook endpoint on Railway validates the `X-Telegram-Bot-Api-Secret-Token` header on every incoming request. Secret token set in @BotFather and stored as a Railway environment variable. Requests without the correct token return 403 immediately.

## Open Questions

1. **Will Moonwell/Avantis pay per verified join?** — Answer this week via DM before the escrow layer is built. If no → revise the advertiser pitch before building step ② payout.
2. **What does Lennox Cartel think about serving an advertiser task in his join flow?** — His reaction = the full KOL objection map. Do this DM before registering any group.
3. **Should the group owner's community ever see who the advertiser is?** ("Sponsored by Moonwell" vs anonymous task) — Default for v1: anonymous task, no advertiser branding in the captcha shown to humans. **Clarification on apparent contradiction with the bid flow:** the bid flow shows competitor names (e.g. "DeFi AI Lab — $0.28") to other advertisers when they are placing a bid. That is advertiser-to-advertiser visibility inside the Canvas AI buy conversation — it is competitive market context, not public branding. The human completing the captcha never sees the advertiser name. These are two separate UX layers. Revisit end-user branding if advertisers request it — some may want "Sponsored by Moonwell" as social proof.
4. **What happens when a group has no active advertiser budget?** — Two distinct empty states:
   - **State A — budget exhausted:** an advertiser exists but `remaining_budget` reached zero. Group owner notified via Canvas AI DM: "Your Canvas AI budget is exhausted — joins are now using generic verification. Top up to resume earning." Advertiser notified separately: "Your campaign for [group] has run out of budget."
   - **State B — no advertiser at all:** group is registered but no active campaign exists. No notification needed. Bot falls back to generic verification silently.
   - In both states: bot falls back to a free generic verification task (no payout, no step ①/②). Group stays gated. State machine exits at PASSED/FAILED as normal.
   - **Generic task format:** open-ended text question stored in `groups.verification_task_text` (set by founder at registration, editable via Canvas AI DM command). Example for DeFi groups: "In one sentence: what do you use DeFi for?" Kimi still scores — quality signal is maintained for the group owner even without a live campaign.
5. **Kimi scoring threshold** — Calibrate from first 50 real responses. Until then, manual review is acceptable for the first campaign. Do not hardcode a threshold before observing real data.
6. **Relayer key custody** — Mitigations defined above (daily spend cap, rate limit, separate key). Full multisig solution deferred to post-launch when escrow values are material.
